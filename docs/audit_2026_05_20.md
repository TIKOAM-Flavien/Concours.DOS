🔴 S-01 — Endpoint debug IP exposé sans auth
Fichier : server/routes/adminRoutes.js:125-135

// Avant
app.get("/api/admin-debug-ip", (req, res) => {
  res.json({ actorIp: ctx.getActorIp(req), ... allowed: Array.from(ctx.adminAllowedIps) });
});

// Après — restreindre à requireAdminAuth + retirer en prod
app.get("/api/admin-debug-ip", requireAdminAuth, (req, res) => {
  if (env.NODE_ENV === "production") return res.status(404).end();
  res.json({ actorIp: ctx.getActorIp(req), reqIp: req.ip });
});

🟠 S-03 — Auth admin par allowlist IP sans password configuré par défaut
Fichier : server/lib/adminAuthHandlers.js:71-75
Risque : Si PORTAL_ADMIN_PASSWORD n'est pas défini, l'admin est ouvert à toute IP allowlistée. Le mode "network-only" est fragile derrière un reverse-proxy mal configuré (trust proxy + X-Forwarded-For spoofing).
Principe violé : Defense-in-Depth
Correctif : Rendre PORTAL_ADMIN_PASSWORD obligatoire en NODE_ENV=production (fail-fast au démarrage), et garder la network-allowlist en complément seulement.

// server/startServer.js (après initDatabase)
if (env.NODE_ENV === "production") {
  const cfg = getAdminAuthConfig(env);
  if (!cfg.passwordConfigured || isWeakAdminPassword(cfg.password)) {
    console.error("[startup] PORTAL_ADMIN_PASSWORD est requis en production");
    process.exit(1);
  }
}

🟠 S-05 — Validation MIME insuffisante (trust-client)
Fichier : server/documentFiles.js:113-167 (info.mimeType du client conservé tel quel)
Risque : Un fichier .exe ou .html renommé .pdf passe le filtrage côté client (accept). À la prévisualisation dans iframe blob, un PDF malveillant pourrait exécuter du JavaScript via les plugins du lecteur PDF.
Correctif : Sniffer la signature binaire avant ACK :

import { fileTypeFromFile } from "file-type"; // package npm
// après écriture sur disque :
const detected = await fileTypeFromFile(storagePath);
if (!detected || !ACCEPTED_MIMES.has(detected.mime)) {
  await unlink(storagePath);
  throw badRequest("Format de fichier non autorisé (signature binaire).");
}

3.2 PERFORMANCE
🟠 P-01 — Téléchargement de fichiers en full-buffer + base64
Fichiers : server/routes/adminRoutes.js:821-836, server/routes/portalRoutes.js:112-138
Impact : Pour un fichier 20 MB, le serveur :
Charge tout en RAM via readFile()
Convertit en base64 (1.33× la taille → ~27 MB)
Le sérialise en JSON → ~50 MB RAM/req, blocage event loop, risque OOM. Le client doit ensuite décoder (atob + boucle for sur 20M itérations dans AdminApp.jsx:35-42).
Avant :


const buffer = await readFile(record.storagePath);
res.json({ fileContent: buffer.toString("base64"), fileName: record.fileName });
Après :


res.download(record.storagePath, record.fileName, {
  headers: { "Cache-Control": "no-store" }
});
// Côté client : utiliser fetch().blob() au lieu de atob()
Gain estimé : −70% RAM serveur, −90% blocage thread client sur fichiers > 5MB.

🟠 P-02 — App.jsx (999 lignes) : god-component
Fichier : src/app/App.jsx
Impact : 26 hooks d'état + 12 useMemo + 6 useEffect. Chaque changement déclenche le re-render du document portal entier (header, KPIs, panel entreprise, grille de cartes, footer). Sur un navigateur lent, ~80 ms par interaction.
Correctif : Extraire <PortalHeader>, <PortalKpis>, <CompanyPanel>, <DocumentGrid> + React.memo.
🟠 P-03 — Décodage base64 synchrone bloquant
Fichier : src/lib/powerAutomateClient.js:43-56, src/app/AdminApp.jsx:35-42
Impact : Boucle for (let i = 0; i < bin.length; i++) sur 20M itérations gèle l'UI plusieurs secondes.
Correctif : Uint8Array.from(atob(...), c => c.charCodeAt(0)) est ~3× plus rapide, ou idéalement passer à un téléchargement binaire :

const res = await fetch(url);
const blob = await res.blob();  // pas de base64 du tout
🟡 P-04 — Pas de lazy-loading des modals admin
Fichier : src/app/AdminApp.jsx:478-537
Impact : ProjectModal, CompanyModal, AdminDocumentReviewModal (~700L au total) sont toujours mountés (open prop). Bundle initial admin alourdi.
Correctif : const ProjectModal = lazy(() => import("./components/admin/ProjectModal")); + <Suspense>.
🟡 P-05 — getAllProjects 2 requêtes + tri JS
Fichier : server/db/repositories/projects.js:9-33
Impact : Charge tous les projets + toutes les sociétés systématiquement. Pour 50 projets × 30 sociétés = 1500 lignes. OK aujourd'hui, mais O(n×m) au tri JS.
Correctif : Un seul LEFT JOIN + agrégation jsonb :

SELECT p.*, COALESCE(jsonb_agg(c.* ORDER BY c."companyName")
       FILTER (WHERE c.id IS NOT NULL), '[]') AS companies
FROM projects p LEFT JOIN companies c ON c."projectId" = p.id
WHERE p."archivedAt" = '' GROUP BY p.id ORDER BY p."createdAt" DESC;
🟡 P-06 — Re-builds inutiles du client Power Automate
Fichier : src/app/App.jsx:235, src/app/AdminApp.jsx:45
Impact : useState(() => createPowerAutomateClient()) recrée correctement seulement à l'init (OK). Mais le client n'a pas de memoization interne pour mimeMap. Mineur.
🟡 P-07 — buildProjectActivityFeed deux passes
Fichier : server/lib/projectActivity.js:373-381
Impact : items.sort(...) puis for...of + if (merged.length >= limit) break; après le sort complet. Pour 1000 items, sort O(n log n) inutile si on veut top 80.
Correctif : Utiliser un heap (priority queue) ou trier après bornage du flux.
🟡 P-08 — Pas de code splitting admin/portal en mode all
Fichier : vite.config.js:51-58
Impact : dist-all/ charge admin+portal dans le même bundle si mode=all. Le visiteur du portail télécharge le code admin.
Correctif : Vite gère naturellement via les input séparés (admin.html, index.html) — vérifier que les chunks partagés ne re-chargent pas inutilement.

L'application Concours.DOS présente un socle technique sain : pas de SQL injection, pas de XSS direct, signatures HMAC correctes, headers de sécurité corrects, séparation des rôles admin/portal.
